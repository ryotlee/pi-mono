在多智能体（Multi-Agent）架构中，记忆系统（Memory System）是决定智能体是“人工智障”还是“贴心助理”的分水岭。对于高校这种长周期、强上下文（从大一入学到大四毕业，或者一个持续数月的科研项目）的场景，传统的线性对话数组（List of Messages）会迅速撑爆大模型的上下文窗口（Context Window），并导致严重的“注意力涣散”和幻觉。

为了支撑 Campus Super Assistant 的统一调度，记忆系统必须做到**分层隔离、动态分配、且支持分支追溯**。

以下是针对该超级助理系统的记忆管理系统深度设计方案：

---

## 一、 核心设计理念：三层级记忆架构

为了兼顾响应速度、Token 成本与数据隐私，系统的记忆池（Memory Pool）被严格划分为三个生命周期不同的层级：

### 1. 全局长期记忆 (Global Long-term Memory / User Profile)

* **定位**：跨会话、伴随用户整个校园生命周期的持久化画像。
* **存储介质**：关系型数据库（如 PostgreSQL）结合向量数据库（Vector DB）。
* **内容结构**：
* **静态标签**：学号、学院、专业、年级、导师姓名（直接从统一认证门户或教务系统同步）。
* **动态偏好**：系统静默总结的用户习惯（例如：“该生倾向于用 Python 进行数据分析”、“该教师通常要求文献摘要不超过 300 字”）。


* **调用逻辑**：Super Agent 每次启动会话时，将这部分信息作为 System Prompt 的前置上下文无感注入。

### 2. 会话状态树 (Session Tree / Medium-term Memory)

* **定位**：单次复杂任务（如：毕业论文开题辅导）的全生命周期记录。
* **存储介质**：基于 `parentId` 链接的 JSONL 结构化文件或图数据库。
* **特点**：摒弃传统的线性对话，采用**有向无环图（DAG）的树状分支结构**。支持用户在漫长的科研探讨中随时“回到上周三的某个节点”开辟新思路。

### 3. 局部工作记忆 (Working Scratchpad / Short-term Memory)

* **定位**：分配给具体 Sub-Agent 的“草稿本”，阅后即焚。
* **存储介质**：内存（Redis 或运行时的内存变量）。
* **工作机制**：当 Super Agent 派发任务给“文档审校 Agent”去阅读一篇 50 页的 PDF 时，这 50 页的内容只存在于该 Sub-Agent 的工作记忆中。任务完成后，Sub-Agent 只向上级返回提炼后的 500 字摘要，随后彻底清空草稿本。这极大保护了主会话的 Token 纯净度。

---

## 二、 底层数据结构：JSONL 与会话分支 (Fork & Tree)

为了支持科研人员和教务人员处理复杂逻辑，系统的对话记录采用独特的树状节点设计。每一个交互回合（Turn）都是 JSONL 文件中的独立一行，通过 `parentId` 维系上下游关系。

**数据结构示例：**

```json
{"id": "msg_001", "role": "user", "content": "帮我构思一个关于图神经网络的选修课大纲。", "parentId": "root"}
{"id": "msg_002", "role": "assistant", "content": "好的，这里有方案A（偏理论）和方案B（偏实践）...", "parentId": "msg_001"}
{"id": "msg_003a", "role": "user", "content": "我们沿着方案A深入，增加公式推导部分。", "parentId": "msg_002"} 
{"id": "msg_003b", "role": "user", "content": "我改变主意了，退回到方案B，重点讲代码实现。", "parentId": "msg_002"} 

```

**业务优势：**

* **多线并发探讨**：教师在备课时，可以在 `msg_002` 这个节点上同时拉出 `msg_003a` 和 `msg_003b` 两个分支（Fork）。模型在处理 `msg_003b` 时，上下文视窗会自动忽略 `msg_003a` 的内容，彻底杜绝交叉污染。
* **无损回溯**：如果新方向走进了死胡同，用户可以一键切换回历史节点，系统只需根据 `parentId` 向上回溯重新拼接 Context 即可。

---

## 三、 记忆紧凑与垃圾回收 (Compaction & GC)

在长达一个学期的项目跟进中，即使是 128K 或 200K 的超大上下文窗口也会捉襟见肘。系统必须具备自动紧凑（Compaction）机制：

### 1. 动态滑动窗口 (Sliding Window)

系统只保留最近 $N$ 轮的原始对话内容（Raw Text），确保当前交流的细节绝对精确。

### 2. 语义浓缩 (Semantic Summarization)

对于超出滑动窗口的旧节点（例如两周前的讨论），系统在后台触发一个低成本的小模型（如 Qwen-7B），将连续的 10 轮对话压缩为一段高度凝练的摘要：“*在 3月1日-3月5日的对话中，用户确定了课题方向为多智能体协同，并排除了强化学习的方案。*” 随后将这段摘要作为单条节点插回树中。

### 3. RAG 归档 (Vectorization)

对于极具价值的长文本（如学生发给系统的长篇实验数据日志），系统在紧凑时，将其切块（Chunking）并转化为向量，存入该会话专属的临时 RAG 库中。当后续对话再次提及这些数据时，通过相似度检索动态调回，而非一直霸占上下文窗口。

---

## 四、 隐私与权限控制 (Privacy & Isolation)

高校数据的敏感度极高（涉及成绩、科研机密、人事档案），记忆系统的分发必须遵守**“最小特权原则 (Principle of Least Privilege)”**。

* **审计与执行分离**：底层维护一个只写不读的 `audit.jsonl`，记录所有原始请求（包含调用时间、IP、完整 Prompt），仅供校级管理员进行安全合规审计。
* **上下文脱敏掩码**：Super Agent 在调度任务时，会充当数据防火墙。如果学生询问“生活 Agent”关于心理咨询室的位置，Super Agent 传递过去的上下文中，会强制剥离该学生的绩点排名和专业信息，防止非相关 Sub-Agent 获取过度画像。

---

通过这种“树状分支 + 动态草稿本 + 语义浓缩”的组合拳，系统既能记住学生大一时的选课习惯，也能在处理几十万字的科研文献时保持敏捷，不会因为记忆过载而崩溃。
